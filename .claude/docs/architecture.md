# Architecture Detail — rrule (@spiandorello/rrulejs)

_Generated by claude-scaffolder. Re-run `/claude-scaffolder` to refresh._

`@spiandorello/rrulejs` is a hardened fork of `rrule` — a JavaScript/TypeScript library implementing RFC 5545 recurrence rules (iCalendar). The fork adds DoS-resistant parsing, iteration guards, TZID caching, and a typed `Recurrence` API with inclusive-day UNTIL semantics.

## Module Breakdown

### `src/index.ts`

Public API entry point. Re-exports `RRule`, `RRuleSet`, `rrulestr`, the typed `Recurrence` API, weekday constants (`MO`..`SU`), and the typed error classes (`RRuleIterationLimitError`, `RRuleStringTooLargeError`). Adding a new public symbol means adding it here.

### `src/rrule.ts`

Core `RRule` class. Owns occurrence retrieval methods (`.all`, `.between`, `.before`, `.after`, `.count`) plus RFC serialization and NLP bridges. Delegates iteration to `iter/` and result accumulation to `IterResult`.

### `src/rruleset.ts`

Composition of multiple rules — `rrule`/`rdate`/`exrule`/`exdate`. Reuses the same `IterResult` backstop so set-level operations are also OOM-safe.

### `src/parsestring.ts` + `src/parseoptions.ts`

Strict RFC 5545 parser. Each RFC key has its own explicitly-typed switch arm (no implicit coercion). Enforces:
- Top-level `maxLength` cap on the input string (default 64 KiB, mutable via `parseStringConfig.maxLength`).
- Per-key numeric validation (`COUNT`, `INTERVAL`, `BYEASTER` reject comma-separated values).
- `BYSETPOS` length capped at 732 entries.
- Throws `RRuleStringTooLargeError` with `actualLength` / `limit`.

### `src/iterresult.ts` + `src/callbackiterresult.ts`

`IterResult` accumulates accepted dates up to `IterResult.defaultMaxIterations` (default 100 000, configurable per-instance). Beyond that it throws `RRuleIterationLimitError`. Applied uniformly across `.all()`, `.count()`, `.between()`, `.before()`, callback iterators, and `RRuleSet`.

### `src/datetime.ts`

Immutable RFC date used by the iterator. `DateTime.add()` enforces `MAX_ADD_ITERATIONS = 100 000` — when hit, calls `markExhausted()` which forces the date past `MAXYEAR=9999` so the outer iterator terminates cleanly. Prevents the parity-mismatch infinite-loop class (e.g. `FREQ=HOURLY;INTERVAL=2;BYHOUR=1`).

### `src/datewithzone.ts` + `src/dateutil.ts`

TZID support. Caches one `Intl.DateTimeFormat` per timezone and memoizes the local TZ (invalidated on `process.env.TZ` change). Brings TZID iteration to ~2× UTC speed (was ~50× slower upstream).

### `src/iter/` + `src/iterinfo/`

State-machine iteration. `iter/index.ts` is the generator. `iterinfo/` holds frequency-specific masks (`monthinfo.ts`, `yearinfo.ts`, deprecated `easter.ts`). Position lists are handled in `iter/poslist.ts`.

### `src/cache.ts`

Optional per-instance result cache, keyed by `(method, args)`. Enabled by `new RRule(options, true)`.

### `src/nlp/`

Bidirectional natural-language layer. `parsetext.ts` parses English-like phrases into RRule options; `totext.ts` produces human-readable strings; `i18n.ts` holds locale strings.

### `src/typedRecurrence/`

Typed `Recurrence` API introduced in 2.9.0. Bridges a string-based `Recurrence` shape (with inclusive-day UNTIL semantics — a calendar-day boundary, not the RFC's exact-instant boundary) to/from RFC `RRule`. `mapper.ts` and `parser.ts` are the conversion entry points; `helpers.ts` formats dates; `constants.ts` holds enum mappings.

## Common Pitfalls

_Static pitfalls only — discoverable from code/config. Runtime lessons live in `.claude/lessons.md`._

**PITFALL:** No top-level rrule string length cap before parsing
**WHY:** A multi-MB string would be fully parsed before any structural validation, consuming memory and CPU.
**INSTEAD:** `parseString()` / `parseInput()` reject inputs longer than `parseStringConfig.maxLength` (default 64 KiB) and throw `RRuleStringTooLargeError`. `parseStringConfig` is exported and mutable for consumers needing a different cap.

---

**PITFALL:** Unbounded `.all()` on infinite rules (no `COUNT`/`UNTIL`) can OOM the process
**WHY:** Rules without `COUNT` or `UNTIL` iterate up to `MAXYEAR=9999`, producing millions of dates.
**INSTEAD:** `IterResult.add()` throws `RRuleIterationLimitError` at `defaultMaxIterations` (100 000, configurable). Applies to `.all()`, `.count()`, `.between()`, `.before()`, callback iterators, and `RRuleSet`.

---

**PITFALL:** Parity-mismatch hang in `HOURLY` / `MINUTELY` / `SECONDLY` (e.g. `FREQ=HOURLY;INTERVAL=2;BYHOUR=1`)
**WHY:** The iteration loop spins until `MAXYEAR` (~35M steps) because `byhour.includes(this.hour)` can never match the current interval phase.
**INSTEAD:** `DateTime.add()` enforces `MAX_ADD_ITERATIONS = 100 000`; on overflow the date is forced past `MAXYEAR` via `markExhausted()`, terminating the iterator. `until` is also propagated into recursive calls.

---

**PITFALL:** Non-positive or non-integer `INTERVAL` causes infinite loops in `DateTime.addHours/Minutes/Seconds`
**WHY:** Upstream had `for(;;)` loops that never terminate when `interval <= 0` or NaN/fractional.
**INSTEAD:** `parseOptions()` validates and throws `Invalid interval: ...` for any non-positive-integer value; missing `INTERVAL` defaults to `1` per RFC.

---

**PITFALL:** O(n²) RFC line unfolding in large payloads
**WHY:** Original `splitIntoLines(unfold=true)` used `splice` inside a `while` loop. N=100k continuation lines took ~3.3 s; N=200k took ~13.5 s.
**INSTEAD:** Single forward-pass rewrite; N=100k now ~5 ms. Only affects `unfold=true` path.

---

**PITFALL:** TZID is ~50× slower than UTC iteration (upstream issue #580)
**WHY:** Every accepted candidate built three fresh `Intl.DateTimeFormat` instances inside `dateInTimeZone()`.
**INSTEAD:** Cache one `Intl.DateTimeFormat` per timezone and memoize local TZ (invalidating on `process.env.TZ` change). `.between()` over a 5-year `DAILY` rule with TZID went from ~374 ms to ~13 ms.

---

**PITFALL:** Bracket-assignment in the RFC option parser allowed implicit type coercion
**WHY:** Original code used `options[optionKey] = value` with `@ts-ignore`, so `COUNT=abc` could be silently stored as a string.
**INSTEAD:** One explicit, typed switch arm per RFC key. `parseNumber()` tightened — non-numeric values throw per-key. `COUNT`, `INTERVAL`, `BYEASTER` reject comma-separated input.

---

**PITFALL:** No `BYSETPOS` array length cap
**WHY:** Legal values are bounded `[-366, 366]`, but `BYSETPOS=1,2,…,10000` was accepted, forcing per-element validation on garbage input.
**INSTEAD:** Cap at 732 entries (count of distinct legal positions: -366..-1, 1..366) and reject longer arrays before per-element validation.

---

**PITFALL:** TypeScript build splits between two `tsconfig` files
**WHY:** `tsconfig.json` is for dev/test (looser, includes test files); `tsconfig.build.json` is production-only (ESM output, declaration maps, narrower includes). Running `tsc` without `-b tsconfig.build.json` will not produce the shipped `dist/esm/` layout.
**INSTEAD:** Always go through `yarn build`, which runs `clean → lint → format-check → tsc -b tsconfig.build.json → webpack`.

---

**PITFALL:** Dual bundle output (UMD + ESM) — easy to ship only one
**WHY:** `dist/esm/` is produced by `tsc`; `dist/es5/` (UMD min + non-min + source maps) is produced by `webpack`. They are independent build steps inside `yarn build`.
**INSTEAD:** Verify both `dist/esm/index.d.ts` and `dist/es5/rrule.min.js` exist after a build before publishing.
