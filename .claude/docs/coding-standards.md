## Naming Conventions

- Identifier denylist: never name a variable, parameter, or property `any`, `Number`, `String`, `string`, `Boolean`, `boolean`, `Undefined`, or `undefined`. Enforced by ESLint `id-denylist`.
- Use named constants for RFC limits and hardening thresholds (e.g. `MAXYEAR`, `MAX_ADD_ITERATIONS`, `IterResult.defaultMaxIterations`, `parseStringConfig.maxLength`) — never inline magic numbers.
- Custom error classes are PascalCase and prefixed `RRule` (e.g. `RRuleIterationLimitError`, `RRuleStringTooLargeError`). Export them from `src/index.ts` so consumers can `instanceof`-check them.
- Test files must use the `*.test.ts` suffix and live under `/test/`, mirroring the source filename (e.g. `src/parsestring.ts` → `test/parsestring.test.ts`).

## Required Patterns

- Strict typing: every value has an explicit or inferable type. `noImplicitAny: true` is set in both `tsconfig.json` and `tsconfig.build.json`.
- Single quotes for string literals; escape only when the string itself contains a single quote (`@typescript-eslint/quotes: ['error', 'single', { avoidEscape: true }]`).
- No semicolons at statement ends — Prettier enforces `semi: false`; ESLint `semi` and `@typescript-eslint/semi` are off so Prettier owns this.
- Two-space indentation, no tabs (`.prettierrc.json`: `tabWidth: 2`, `useTabs: false`).
- `===` / `!==` only; `==` / `!=` is allowed solely when comparing against `null` (`eqeqeq: ['error', 'smart']`).
- `const` / `let` only — `var` is forbidden (`no-var: error`).
- One declaration per `const`/`let` statement (`one-var: ['error', 'never']`).
- 1TBS brace style: opening brace on the same line as the statement (`brace-style: ['error', '1tbs']`).
- `curly: ['error', 'multi-line']` — wrap multi-line bodies in braces; single-line bodies may omit them.
- Every file ends with a newline (`eol-last: error`); no trailing spaces (`no-trailing-spaces: error`); no consecutive blank lines (`no-multiple-empty-lines: error`).
- Always pass `radix` to `parseInt` (`radix: error`).
- Use `new Foo()` with parentheses, never `new Foo` (`new-parens: error`).
- `throw new Error(...)` — never throw string or other literals (`no-throw-literal: error`).
- Type-only references to ambient declarations must use `import` rather than triple-slash `/// <reference types=...>` (`@typescript-eslint/triple-slash-reference: { types: 'prefer-import' }`).
- Awaited expressions must be thenables (`@typescript-eslint/await-thenable: error`).
- Every Promise must be awaited, `.then`-chained, or explicitly voided with `void` (`@typescript-eslint/no-floating-promises: error`).
- Use type assertions with the `as` form, consistently (`@typescript-eslint/consistent-type-assertions: error`); do not assert when the cast is provably redundant (`@typescript-eslint/no-unnecessary-type-assertion: error`).
- Member delimiters in multi-line type/interface bodies: no trailing delimiter on each line, but final member must end the block cleanly (`@typescript-eslint/member-delimiter-style` multiline: `none`/`requireLast: true`; singleline: `semi`/`requireLast: false`).
- JSDoc on exported APIs: align asterisks, indent consistently, and add a blank line after the description (`jsdoc/check-alignment`, `jsdoc/check-indentation`, `jsdoc/newline-after-description`).
- Use named imports from local modules; never re-import the same module twice (`no-duplicate-imports: error`).
- Do not import deprecated APIs (`import/no-deprecated: error`).
- Validate user input at the parser/option boundary and throw a typed error class (mirror the patterns in `src/parsestring.ts` and `src/parseoptions.ts`).
- Guard every unbounded iteration with an explicit cap (`MAX_ADD_ITERATIONS` in `DateTime.add()`, `defaultMaxIterations` in `IterResult.add()`). When the cap trips, surface an `RRuleIterationLimitError` or call `markExhausted()` — never spin silently.

## Forbidden Patterns

<!-- override: repo bans `any` outright (no escape-comment carve-out) -->
- No `any` type anywhere. `@typescript-eslint/no-explicit-any: error` has no override — refactor to a real type, `unknown`, or a narrow generic.
- No `eval` and no `arguments.caller` / `arguments.callee` (`no-eval: error`, `no-caller: error`).
- No empty functions or empty blocks (`no-empty: error`, `no-empty-function: error`, `@typescript-eslint/no-empty-function: error`). If a no-op is intentional, add a comment explaining why and use a named helper.
- No assignment inside conditions (`no-cond-assign: error`).
- No constant conditions in `if`/`while`/ternaries (`no-constant-condition: error`). The historical `for(;;)` patterns in `DateTime.add*` are exactly what this catches — replace them with bounded loops.
- No control characters in regex literals (`no-control-regex: error`); no whitespace runs inside regex (`no-regex-spaces: error`); invalid regex literals are errors (`no-invalid-regexp: error`).
- No fallthrough in `switch` without an explicit comment (`no-fallthrough: error`).
- No redeclared identifiers (`no-redeclare: error`).
- No unused labels (`no-unused-labels: error`).
- No `return await x` (`no-return-await: error`) — return the promise directly.
- No `NaN` comparisons; use `Number.isNaN` (`use-isnan: error`).
- No `new` / `constructor` misuse on classes (`@typescript-eslint/no-misused-new: error`).
- No unnecessary namespace/enum qualifiers (`@typescript-eslint/no-unnecessary-qualifier: error`).
- No overloaded signatures that could be merged with a union (`@typescript-eslint/unified-signatures: error`).
- No unused expressions, except tagged-template calls and short-circuit/optional-chaining for side effects (`@typescript-eslint/no-unused-expressions: ['error', { allowTaggedTemplates: true, allowShortCircuit: true }]`). This rule is disabled inside `*.test.*` files so `expect(...).to.equal(...)`-style assertions are allowed.
- Do not introduce new `console.*` calls or other debugging artifacts in committed source (`eslint:recommended` baseline; reviewers should reject leftover debug output).
- Do not bypass the parser's strict typing with `@ts-ignore` or bracket-keyed option assignment in `parseString` / `parseOptions`. Add a typed `switch` arm and reject invalid input with a per-key error instead (see Pitfall in `CLAUDE.md`).
- Do not regress the O(N) line-unfolder in `rrulestr.ts` back into a `splice`-inside-`while` shape — it must stay a single forward pass.
- Do not construct fresh `Intl.DateTimeFormat` per candidate in `datewithzone.ts` / `dateutil.ts` — reuse the memoized per-TZID instance.

## Testing Requirements

- Test runner: Jest with `ts-jest` preset, `testEnvironment: 'node'` (`jest.config.js`).
- Every `src/**/*.ts` module with public behaviour must have a matching `test/<name>.test.ts`. New error classes and new guards must include a dedicated test (see `test/iterresult.test.ts` and `test/parsestring.test.ts` as patterns).
- DoS-guard tests must cover both the happy path and the limit-trip path (e.g. asserting that `RRuleIterationLimitError` is thrown, that `actualLength`/`limit` are populated on `RRuleStringTooLargeError`, and that `markExhausted()` short-circuits cleanly).
- Use the `datetime()` helper from `dateutil.ts` for UTC fixtures (1-based months) instead of `new Date(...)` literals.
- Use `mockdate` for tests that depend on "now". Reset it in `afterEach` to keep tests isolated.
- Tests must remain green under all CI timezone matrices: `America/Vancouver`, `America/Los_Angeles`, `Africa/Nairobi`, `Pacific/Kiritimati`, `Asia/Tokyo`. If a test relies on a specific TZ, set it explicitly inside the test rather than assuming the runner default.
- Coverage runs via `yarn test-ci` (`nyc` + Jest); CI excludes `.d.ts` from coverage. Do not add ad-hoc `/* istanbul ignore */` comments to dodge coverage — fix the missing test.

## TypeScript Build

- Two TS configs:
  - `tsconfig.json` — development + test compilation. `noEmit: true`, `module: commonjs`, `target: es6`, includes both `src/` and `test/`.
  - `tsconfig.build.json` — production build. Emits to `dist/esm/`, `declaration: true`, `declarationMap: true`, `module: es2015`, `target: es5`, `importHelpers: true` (depends on `tslib`).
- Both configs set `noImplicitAny: true`, `noEmitOnError: true`, `sourceMap: true`, `esModuleInterop: true`. Do not relax these.
- Production build excludes `test/**/*` — keep test-only helpers out of `src/` (or guard them so they tree-shake out).
- UMD bundle is produced by Webpack (`webpack.config.js`) with `library: 'rrule'`, `libraryTarget: 'umd'`, Terser-minified plus an unminified twin via `unminified-webpack-plugin`. Source maps are always emitted (`devtool: 'source-map'`).
- Public API surface lives in `src/index.ts`. Anything not re-exported there is internal and may change without a semver bump.

## Security Hardening (project-specific)

This fork's reason for existing is hardening against DoS-class issues in upstream `rrule`. Any change touching parsing or iteration must preserve these invariants — see `SECURITY.md` for the authoritative threat model and rationale.

- Iteration cap: `IterResult.defaultMaxIterations` (100_000) bounds `.all()`, `.between()`, `.before()`, `.after()`, callback iterators, and `RRuleSet`. Throws `RRuleIterationLimitError` on overflow. The cap is mutable but lowering it must not silently truncate results.
- DateTime add cap: `MAX_ADD_ITERATIONS = 100_000` in `DateTime.add()` prevents parity-mismatch hangs on `HOURLY`/`MINUTELY`/`SECONDLY` rules. On trip, the DateTime is forced past `MAXYEAR` via `markExhausted()`.
- Interval validation: `parseOptions()` rejects non-positive or non-integer `interval` with `Invalid interval: ...`. Missing interval defaults to `1` per RFC.
- String length cap: `parseString()` / `parseInput()` reject inputs longer than `parseStringConfig.maxLength` (default 64 KiB, mutable from package root) with `RRuleStringTooLargeError` carrying `actualLength` and `limit`.
- BYSETPOS length cap: at most 732 entries (the count of legal positions `-366..-1, 1..366`); reject longer arrays before per-element validation.
- Line unfolding: the `splitIntoLines(unfold=true)` path must stay O(N) (single forward pass). Do not reintroduce the upstream `splice`-in-`while` shape.
- Timezone caching: keep one cached `Intl.DateTimeFormat` per TZID in `datewithzone.ts`/`dateutil.ts`, and memoize the local TZ with invalidation on `process.env.TZ` changes.
- Typed option parsing: in `parseString.ts` / `parseOptions.ts`, every RFC key gets its own `switch` arm with explicit typed assignment. `parseNumber()` throws a per-key error on non-numeric input. `COUNT`, `INTERVAL`, and `BYEASTER` reject comma-separated input. Do not regress to bracket-keyed assignment under `@ts-ignore`.

## Typed Recurrence API (new in 2.9.0)

- Live under `src/typedRecurrence/`; the public surface (`Recurrence`, `RecurrenceFrequency`, `recurrenceToRRule`, `parseRecurrence`, etc.) is re-exported from `src/index.ts`.
- `Recurrence.until` uses inclusive-day semantics — when bridging to RFC `UNTIL`, normalise to the end of that local day. Tests in `test/typedRecurrence.test.ts` pin this behaviour; do not change semantics without updating both directions of `mapper.ts` and the test fixtures.
- Conversions must round-trip: `recurrence → RRule string → recurrence` should be lossless for the documented field set.

## Linter & Formatter Operation

- Format with Prettier (`yarn format` writes; `yarn format-check` verifies). CI runs `format-check` and fails on diff.
- Lint with `yarn lint` (`eslint . --fix --config .eslintrc.js`). Build fails if lint reports errors.
- Pre-commit hook (`.husky/pre-commit`) runs `lint-staged`, which executes `yarn lint` then `yarn format` on staged `*.ts` files. Do not bypass with `--no-verify` without a stated reason.
- ESLint and Prettier ignore lists:
  - ESLint ignores: `dist/`, `coverage/`, `webpack.config.js`, `.eslintrc.js`, `jest.config.js`.
  - Prettier ignores: `node_modules/`, `bower_components/`, `.idea`, `.vscode`, `.claude`, `*.log`, `coverage`, `.nyc_output`, `dist/`.
- The build command (`yarn build`) is the canonical local CI: `lint → format-check → tsc -b tsconfig.build.json → webpack → tsc on emitted d.ts`. Run it before opening a PR.
