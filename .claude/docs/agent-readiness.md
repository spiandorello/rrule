## 1. AI Context Files & Documentation

### Strengths

- `CLAUDE.md` at repo root (12.2 KB) — primary agent context file.
- `.claude/docs/architecture.md` (12.2 KB) — extended architecture documentation.
- `.claude/docs/coding-standards.md` (12.8 KB) — formal coding standards reference for agents.
- `SECURITY.md` (5.8 KB) — DoS-focused threat model with severity-ordered pitfalls, code references, and rationale (authoritative source for hardening decisions).
- `README.md` (28.9 KB) — API surface, usage examples, timezone semantics, RFC differences, migration guide from upstream.
- `CHANGELOG.md` (7.5 KB) — version history including the 2.9.0 Typed Recurrence API release.
- `.claude/lessons.md` (948 B) and `.claude/extract-lessons-prompt.md` (1.7 KB) — captured lessons from prior agent sessions.

### Gaps

- No `CONTEXT.md` or domain-language glossary file at repo root — agents must infer terminology (RFC 5545 keywords, `BYDAY` vs `byweekday` rename, inclusive-day UNTIL semantics) from `CLAUDE.md` and `README.md` rather than a single canonical glossary.
- No `docs/adr/` (Architecture Decision Records) directory — major hardening decisions (100 k iteration cap, 64 KiB string length cap, 732-entry BYSETPOS cap, TZID Intl cache) are explained in `SECURITY.md` prose but not formalised as ADRs.
- No `.env.example` (not strictly needed — library has no runtime env vars — but CI does depend on `LANG` and `TZ` which are not documented as a local-dev expectation).

## 2. Coding Standards & Enforcement

### Strengths

- ESLint config (`.eslintrc.js`) with `@typescript-eslint/no-explicit-any: error`, `@typescript-eslint/no-floating-promises: error`, `@typescript-eslint/no-empty-function: error`, and an `id-denylist` forbidding `any, Number, String, string, Boolean, boolean, Undefined, undefined`.
- Prettier (`.prettierrc.json`) — `semi: false, singleQuote: true, tabWidth: 2`.

- CI step `yarn build` runs `yarn clean && lint && format-check && tsc && webpack` — every PR must pass lint + format-check + tsc.
- `.claude/docs/coding-standards.md` (12.8 KB) — agent-readable companion to the lint config.

### Gaps

- No `commitlint` / commit-message convention enforced — recent commits follow Conventional Commits informally (`chore(format):`, `feat(typed-recurrence):`) but nothing in CI or Husky validates this.

- README mentions "JavaScript Standard Style" but the actual enforced style is Prettier with `semi: false` — minor inconsistency.

## 3. Test Infrastructure

### Strengths

- Jest 29 with `ts-jest` preset; `testEnvironment: 'node'`; 14 test files in `/test/`.
- Comprehensive coverage of DoS guards: `iterresult.test.ts` (RRuleIterationLimitError 100 k cap), `datetime.test.ts` (MAX_ADD_ITERATIONS hang guard), `parsestring.test.ts` (RRuleStringTooLargeError), `datewithzone.test.ts` (TZID perf regression).
- Tests run in 4 TZ/LANG contexts in CI: `America/Vancouver`, `America/Los_Angeles`, `Africa/Nairobi`, `Pacific/Kiritimati`, plus `Asia/Tokyo` with coverage.
- `mockdate` (`^3.0.5`) available for frozen-time tests.
- `datetime()` helper in `dateutil.ts` for constructing UTC dates with 1-based months.
- Custom error classes (`RRuleIterationLimitError`, `RRuleStringTooLargeError`) tested in isolation.
- `.claude/hooks/pre-push.js` runs `npx jest --findRelatedTests` on changed `*.ts`/`*.tsx` files before pushing — agents get a fast regression check.

### Gaps

- No coverage threshold enforced. `codecov.yml` reports coverage but `jest.config.js` does not set `coverageThreshold` and CI does not fail on coverage drops.
- No E2E setup — acceptable for a pure library, but means `between()`/`all()` integration with downstream consumers (calendar UIs, ICS pipelines) is not tested.
- No documented factories/builders for `Options` objects — agents writing new tests must hand-construct Options or copy from existing tests.
- No mutation testing or property-based testing (e.g. fast-check) — given the parser's adversarial input surface and the DoS guards, property tests would harden the suite further.

## Recommendations

### High Priority (critical gaps blocking agent effectiveness)

1. **Configure GitHub MCP** — add a `github` server to `.mcp.json` and enable it in `.claude/settings.json` `enabledMcpjsonServers`. Without it, the `create-pr` skill must shell out to `gh` for every PR/issue interaction; GitHub MCP gives the agent structured access to PR state, CI runs, and review comments. This is the largest remaining gap for ticket-to-PR workflows.
2. **Populate or remove empty `.claude/commands/` subdirectories** — `create-pr/`, `create-skill-or-rule/`, `linear-ticket-reviewer/`, and `rrule-hardening/` are empty directories. The corresponding content already exists under `.claude/skills/<name>/SKILL.md`. Either delete the empty command directories (they shadow nothing useful) or add slash-command entry points (`.claude/commands/<name>.md`) that delegate to the matching skill.
3. **Enforce a Jest coverage threshold in CI** — add `coverageThreshold` to `jest.config.js` (e.g. statements/branches/lines/functions ≥ 80%) so coverage regressions fail the build instead of just being reported to codecov. The DoS-guard tests are the load-bearing safety net; silently shrinking them is a real risk.

### Medium Priority

4. **Add a CODEOWNERS file** — route PRs touching `src/parsestring.ts`, `src/iterresult.ts`, `src/datetime.ts`, and `SECURITY.md` to an explicit reviewer (the security-hardening surface).
5. **Add Dependabot or Renovate config** — `.github/dependabot.yml` would automate weekly dependency PRs against a repo whose entire value proposition is "hardened fork". Stale transitive deps undermine that.
6. **Add a CodeQL workflow** — `.github/workflows/codeql.yml` for static security scanning. Free for public repos and complements the runtime DoS guards.
7. **Document local TZ/LANG expectations in `CLAUDE.md` or `README.md`** — note that CI runs tests in 4 timezones and that local-only `yarn test` may miss TZ-sensitive regressions. Suggest `TZ=America/Vancouver yarn test` as a quick sanity check.
8. **Add an `engines` field to `package.json`** — pin Node ≥ 14 (matching the CI floor) to fail fast on unsupported runtimes during `yarn install`.
9. **Promote SECURITY.md decisions into formal ADRs** — create `docs/adr/0001-iteration-cap.md`, `0002-string-length-cap.md`, `0003-bysetpos-cap.md`, `0004-tzid-intl-cache.md`. ADRs are agent-friendly (each is short, dated, and links cause→decision→consequence).

### Low Priority (nice to have)

10. **Add a `yarn dev` / watch mode** — e.g. `"dev": "tsc -b tsconfig.build.json --watch"` for faster iteration on the iterator and parser.
11. **Add commit-message linting** — `@commitlint/config-conventional` via Husky `commit-msg` hook, since the existing commit log already follows Conventional Commits informally.
12. **Add property-based tests** — `fast-check` over parser inputs (random RRULE strings under 64 KiB) and iterator inputs (random `Options` configurations) would harden the DoS guards beyond the current example-based tests.
13. **Configure Chrome DevTools MCP** — only if a docs/playground surface is added later. Not needed for the current library-only scope.
14. **Add a `CONTRIBUTING.md`** — distill the workflow currently scattered across `CLAUDE.md`, `.claude/commands/workflow.md`, and the skills into a single human-friendly contributor doc.
